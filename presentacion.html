Invoice Reminder System

Simulás un sistema de gestión de facturas: detectás vencimientos y enviás recordatorios de pago.
Tu misión: extender el backend.

Node.js ≥ 18 Express REST API Git + PR 4 Tareas

📋 ¿De qué se trata?

El backend base ya está implementado. Tu trabajo es extenderlo con endpoints nuevos, validaciones y manejo de errores correcto.

✓ Lo que ya existe

  • GET /invoices — listar facturas
  • POST /invoices — crear factura

✗ Lo que falta (tus tareas)

  • ▸ Detectar facturas vencidas
  • ▸ Enviar recordatorios de pago
  • ▸ Validaciones de entrada
  • ▸ Códigos HTTP correctos
💡 No buscamos algoritmos complejos. Queremos ver cómo razonás, cómo organizás tu código y cómo tomás decisiones técnicas simples pero correctas.

🗂️ Modelo de datos

Cada factura tiene la siguiente forma:

JSON{
  "id":            1,
  "clientName":   "Empresa ACME",
  "amount":        120000,
  "dueDate":       "2026-02-10",   // fecha de vencimiento
  "status":        "pending",      // "pending" | "paid"
  "reminderSent": false            // ¿se envió recordatorio?
}
⚠️ Clave conceptual: una factura puede ser pending sin estar vencida. Está vencida cuando status === "pending" y dueDate es anterior a hoy.

1️⃣ Tarea 1 — GET /invoices/overdue Endpoint nuevo

Una factura está vencida cuando cumple ambas condiciones:

  • status es "pending"
  • dueDate es anterior a la fecha actual
📌 Responder 200 con el array (puede ser vacío).

Respuesta esperada 200

JSON[
  {
    "id": 1,
    "clientName": "Empresa ACME",
    "amount": 120000,
    "dueDate": "2026-01-20",
    "status": "pending",
    "reminderSent": false
  }
]

2️⃣ Tarea 2 — POST /reminders/:invoiceId Endpoint nuevo

El endpoint debe:

1
Verificar que la factura exista → Responder 404 si no se encuentra
2
Marcar reminderSent = true en la factura correspondiente
3
Simular el envío → console.log del recordatorio

Respuesta esperada 200

JSON{
  "message": "Recordatorio enviado
            a Empresa ACME",
  "invoice": { ... }
}
⚠️ Sugerencia: Revisá el código existente con atención. Puede haber comportamientos inesperados.

3️⃣ Tarea 3 — Validaciones en POST /invoices Validaciones

Validar antes de crear la factura:

  • clientName es obligatorio
  • amount debe ser mayor a 0
  • dueDate debe ser una fecha válida
💡 Mensajes de error claros y específicos para el usuario.

Respuesta esperada 400

JSON{
  "error": "El campo clientName
          es obligatorio"
}

Si todo está bien 201

JSON{
  "id": 11,
  "clientName": "Nueva SA",
  ...
}

4️⃣ Tarea 4 — Manejo de errores HTTP Revisión

Revisá todos los endpoints existentes y nuevos. Asegurate de usar siempre el código correcto:

Código Cuándo usarlo
200Operación exitosa
201Recurso creado correctamente
400Error de validación (datos incorrectos del cliente)
404Recurso no encontrado
500Error interno del servidor
